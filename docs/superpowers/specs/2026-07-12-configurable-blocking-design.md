# QQ 音乐可配置屏蔽设计

日期：2026-07-12

## 目标

为 `QQMusic.AggressiveCleaner.plugin` 增加 Loon 原生配置开关，使用户能独立控制不同类型的屏蔽功能。所有开关首次安装时默认开启，保持当前“激进清理”的产品定位；关闭某一开关只停止该类别的处理，不影响其他类别。

播放、搜索、登录、歌词、歌曲、歌手、专辑及歌单等核心功能必须始终受保护。

## 抓包结论

最新 Loon 归档包含 8034 个会话，其中 234 个请求命中 `u6.y.qq.com/cgi-bin/musics.fcg`。

上一版请求脚本删除 `m-encoding: m1` 并改用 gzip 后，共有 136 个原生 QQ 音乐请求经过头部替换，这些请求只收到 `code: 500001`。因此该响应协商方案不可用，必须撤回，后续脚本不得修改 `m-encoding` 或 `accept-encoding`。

抓包同时证明：

- 直播推荐、标签气泡、广告能力检测等请求通常是独立模块，可在请求侧安全拦截。
- 部分请求把广告、视频、歌单扩展信息等模块聚合在同一二进制请求中，不能因为命中一个屏蔽模块就中断整个请求。
- WebView 发起的 `musics.fcg` 请求可返回 Loon 已解压的 JSON，适合在响应侧按字段和节点清理。
- 未知二进制及不可解析响应必须原样放行。

## Loon 配置

插件通过 `[Argument]` 声明以下 `switch`，默认值全部为 `true`：

| 参数 | Loon 显示名称 | 控制范围 |
| --- | --- | --- |
| `block_splash` | `[广告] 屏蔽开屏广告` | 开屏素材、开屏策略和启动广告 |
| `block_popup` | `[广告] 屏蔽弹窗与气泡` | 弹窗、浮层、标签气泡 |
| `block_banner` | `[广告] 屏蔽横幅与运营卡片` | Banner、运营卡片、推广入口 |
| `block_live` | `[入口] 屏蔽直播` | 直播入口、直播标签和直播推荐 |
| `block_video` | `[入口] 屏蔽视频与短视频` | 视频、短视频入口和相关推广内容 |
| `block_mall_activity` | `[入口] 屏蔽商城、活动与福利` | 商城、活动、福利、抽奖和装扮入口 |
| `block_promo_recommend` | `[推荐] 屏蔽推广推荐内容` | 带推广性质的推荐位，不删除普通歌曲推荐 |
| `block_telemetry` | `[隐私] 屏蔽埋点与上报` | 曝光、追踪、报告和广告归因请求/字段 |

这些参数以 `argument=[...]` 同时传给请求脚本和响应脚本。脚本内部将字符串或布尔值统一转换为布尔值；参数缺失时按 `true` 处理，以兼容已有安装和保持默认激进策略。

## 架构

### 请求控制器

保留一个请求脚本，职责改为“判断是否安全中断请求”，不再修改任何请求头。

控制器读取 URL，并在启用 `binary-body-mode=true` 时扫描二进制请求体中可见的模块标识。它先为命中内容分类，再判断请求是否只包含可屏蔽模块：

- 独立广告、弹窗、直播、活动或上报请求：对应开关开启时中断。
- 同一请求同时含有播放、歌曲、歌单、搜索、登录、歌词、会员状态等核心模块：无条件放行。
- 无法可靠识别、模块不完整或存在类别冲突：放行。

“无法证明可以安全屏蔽”统一视为放行，避免破坏核心链路。

### 响应清理器

现有 JSON 树清理器改为按类别决策。每个字段、节点、文本和 URL 先被分类，再检查对应开关；关闭的类别不参与删除。

清理器继续执行以下保护：

- 包含歌曲、音轨、歌手、专辑、歌单等稳定身份字段的核心音乐对象整体保留。
- 根对象不得被清空或改变顶层类型。
- JSON 解析失败、输入为空、二进制输入或序列化失败时返回空修改对象，让 Loon 原样放行。
- 普通音乐推荐不因 `recommend` 或 `feed` 字段名被删除；只有同时具备推广信号时才归入 `block_promo_recommend`。

### URL Rewrite 调整

现有不可配置的静态 `reject` 规则将被移除或收敛，避免用户关闭开关后仍被静态规则拦截。同类阻断交由请求控制器统一执行，确保 Loon UI 开关是唯一有效配置来源。

## 分类优先级

分类按以下顺序执行，首个明确类别生效：

1. 核心音乐保护
2. 直播
3. 视频/短视频
4. 商城/活动/福利
5. 开屏广告
6. 弹窗/气泡
7. 横幅/运营卡片
8. 推广推荐
9. 埋点/上报

例如 `MliveRecommend...` 归类为直播，而不是普通推荐；视频歌单混合请求因核心保护而放行。

## 错误处理与兼容性

- 不再协商或转换 QQ 音乐的 `m1` 编码。
- 请求体不是 `Uint8Array` 时仍支持字符串扫描；两者都不可用时仅依据 URL 判断。
- `$argument` 不存在或格式异常时使用全开默认值。
- Loon 脚本入口出现异常时调用 `$done({})` 放行，而不是返回空响应。
- 配置要求 Loon build 733 或更高版本，因为 `[Argument]` 自该版本起受支持。

## 测试策略

测试采用最小、去敏后的抓包结构，不提交原始抓包文件。

必须覆盖：

1. Manifest 声明八个默认开启的 switch，并把参数传给两个脚本。
2. 请求脚本不修改 `m-encoding` 或 `accept-encoding`。
3. 每个类别在开关开启时被屏蔽，关闭时原样放行。
4. 直播推荐优先归类为直播，不受推广推荐开关替代。
5. 含广告/视频与歌单核心模块的混合请求必须放行。
6. 普通歌曲推荐在推广推荐开关开启时仍保留。
7. 非 JSON、未知二进制和异常参数安全放行。
8. 所有开关缺失时按全开处理。

## 验收标准

- Loon 插件详情页显示八个可独立操作的开关，默认全部开启。
- 关闭任一开关并刷新插件后，对应类别停止屏蔽，其他类别保持原行为。
- 原生 `musics.fcg` 请求不再因头部修改返回 `500001`。
- QQ 音乐的播放、搜索、登录、歌词和歌单浏览能够正常工作。
- 自动化测试全部通过，原始抓包及任何账号标识不进入 Git 历史。
